# LangSAM 接口调用问题记录

## 1. 一开始误以为只能通过页面调用

刚开始看这个项目时，首先看到的是 `app.py` 里的 Gradio 页面，所以第一反应是这个项目只能通过网页来使用。也就是先上传图片，再输入 prompt，然后点击按钮，最后返回结果图。

这个理解从使用角度看没有问题，但如果要写接口文档，就不能只停留在页面层面。继续看 `lang_sam/server.py` 和 `lang_sam/lang_sam.py` 后发现，Gradio 页面只是把调用流程包装了一层，真正执行模型推理的是 `LangSAM.predict()`。

所以这里需要先把项目分成三层来看：

1. `app.py`：主要负责页面交互；
2. `server.py`：负责接收请求并调用模型；
3. `lang_sam.py`：负责真正的模型推理。

把这几层关系理清之后，才比较适合继续整理后端接口和终端调用方式。

## 2. 外层输入是图片路径，模型实际接收的是图片对象

从后端接口的角度看，最方便的输入一般是图片路径和文本 prompt。例如：

- `image_path`
- `text_prompt`

查看代码后发现，`LangSAM.predict()` 本身并不是直接接收图片路径，而是接收 `PIL.Image` 对象组成的列表。

调用形式大致如下：

```python
results = model.predict(
    images_pil=[image_pil],
    texts_prompt=[text_prompt],
    box_threshold=0.3,
    text_threshold=0.25,
)
```

因此在整理接口时，我把输入分成了两层：

- 外层接口输入：图片路径、文本 prompt；
- 模型实际输入：`PIL.Image` 列表、prompt 列表。

这样写清楚后，后端成员就不会误以为模型函数可以直接传入 `image_path`。实际流程应该是先根据图片路径读取图片，再把图片对象传给模型。

## 3. 原项目没有直接可用的终端调用脚本

原项目已经有前端页面和服务接口，但它没有提供一个比较直接的命令行脚本来完成完整流程。比如下面这种流程，原来不是特别方便直接演示：

1. 输入图片路径；
2. 输入文本 prompt；
3. 调用 LangSAM；
4. 保存 mask 和结果图；
5. 输出结构化结果文件。

如果只展示 Gradio 页面，虽然能说明模型跑通了，但不太方便说明“后端如何直接调用模型”。所以后面我补充了一个 `cli_predict.py`，把原来的页面调用逻辑整理成了终端版本。

整理后的调用链条可以写成：

```
image_path + prompt → 读取图片 → 调用 LangSAM.predict() → 保存输出结果 → 后端读取结果文件
```

这样后端接入时会更清楚一些。

## 4. 初始脚本导入顺序不太合理

一开始写 `cli_predict.py` 时，我把 `lang_sam`、`numpy`、`PIL` 这些依赖都放在文件开头导入。后来发现这样有一个问题：即使只是想查看帮助信息，比如执行：

```powershell
python .\cli_predict.py --help
```

程序也会先加载模型相关依赖，而不是马上显示参数说明。这样体验不太好，也不方便排查参数问题。

后面调整为：

- `argparse` 保留在前面；
- `numpy`、`PIL`、`lang_sam` 等重型依赖放到 `main()` 里面；
- 只有真正执行预测时才加载模型依赖。

这样修改后，查看 `--help` 时不会再卡在模型加载上。

## 5. 第一次运行时容易把“加载慢”看成“程序失败”

运行 `cli_predict.py` 时，LangSAM 会加载 `transformers`、GroundingDINO、SAM 等相关依赖，所以第一次启动会比较慢。刚开始终端长时间没有明显输出，我也容易误以为程序卡住了。

为了让运行过程更清楚，后面在脚本里加了一些提示信息，例如：

```text
LangSAM CLI started.
Loading LangSAM dependencies...
Initializing model...
Running prediction...
Saving outputs...
```

加上这些日志后，就能大概知道程序现在运行到哪一步。如果是正在初始化模型，就不需要马上判断为报错；如果卡在保存输出，也能更快定位问题。

## 6. 调整导入位置后出现了新的小问题

把重型依赖移到 `main()` 之后，虽然解决了 `--help` 加载慢的问题，但也出现了新的作用域问题。

比如 `save_outputs()` 函数里原本直接用了：

```python
Image.fromarray(...)
```

但因为 `Image` 不再是全局导入，所以运行时出现了：

```text
NameError: name 'Image' is not defined
```

这个问题说明，改脚本结构时不能只改导入位置，还要检查其他函数有没有依赖这些全局对象。最后我把 `Image` 显式传进 `save_outputs()`，避免函数偷偷依赖外部全局变量。

## 7. 原始输出不太方便后端继续处理

原项目中，模型结果主要有两种形式：

1. `LangSAM.predict()` 返回内存中的结果字典；
2. `server.py` 把可视化结果作为 HTTP 图片响应返回。

这两种方式都能完成推理，但对我们这个项目来说还不够方便。因为后端还需要继续读取 mask，生成透明 PNG，并把结果路径返回给前端。如果结果只在内存里，或者只作为 HTTP 响应返回，后续处理就不太直观。

因此，我在终端调用脚本里增加了结果落盘逻辑，把输出统一保存为：

- `mask.png`
- `result.png`
- `result.json`

这样后端后续只需要读取 `result.json` 和对应图片文件，就可以继续做 PNG 生成和前端返回，不需要重新拆模型输出对象。

## 8. 接口调用问题汇总表

| 问题类型           | 具体表现                                     | 处理方式                                         |
| ------------------ | -------------------------------------------- | ------------------------------------------------ |
| 调用入口理解不清   | 一开始容易以为只能通过 Gradio 页面调用       | 梳理 `app.py`、`server.py`、`lang_sam.py` 的关系 |
| 输入层次混淆       | 容易把 `image_path` 误写成模型函数的直接输入 | 区分外层接口输入和模型实际输入                   |
| 缺少终端调用方式   | 不方便展示后端如何直接调用模型               | 补充 `cli_predict.py`                            |
| `--help` 加载慢    | 查看帮助时也会加载模型依赖                   | 参数解析前置，模型依赖延后导入                   |
| 启动过程不清楚     | 模型加载慢时容易误以为卡死                   | 增加阶段性日志                                   |
| 导入顺序调整后报错 | 后移导入后出现 `Image` 未定义                | 将依赖对象显式传入函数                           |
| 输出不方便复用     | 原始结果主要是内存对象或 HTTP 响应           | 增加结果落盘，保存图片和 JSON                    |
